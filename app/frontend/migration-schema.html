📦 Migration Schema — Destination Database

Reference for building the client-facing product database. Source: arbt.ly (PostgreSQL + pgvector).

Overview

What this page is: A self-contained reference for an agent (or developer) building the destination database that receives product data from ArbitLens.

What migrates: 14 product columns (core data) + optional matches table (denormalized).

What stays at ArbitLens: 5 infrastructure columns (id, image_hash, embedding, created_at, last_updated) — these power scraping and matching but aren't product data.

Sync pattern: Delta push — we compare our last_updated against your synced_at and push only changed rows.

CREATE TABLE

CREATE TABLE products ( id SERIAL PRIMARY KEY, source_key VARCHAR(150) UNIQUE NOT NULL, platform VARCHAR(50) NOT NULL, platform_id VARCHAR(100) NOT NULL, title TEXT, price NUMERIC(10,2), currency VARCHAR(10), url TEXT, image_urls TEXT[], sales_30d INTEGER DEFAULT 0, review_count INTEGER DEFAULT 0, review_avg NUMERIC(3,1), category_l1 VARCHAR(100), category_l2 VARCHAR(100), category_l3 VARCHAR(100), is_active BOOLEAN DEFAULT TRUE, synced_at TIMESTAMP DEFAULT NOW() ); -- Indexes CREATE INDEX idx_dest_platform ON products (platform); CREATE INDEX idx_dest_category ON products (category_l1, category_l2, category_l3); CREATE INDEX idx_dest_sales ON products (sales_30d DESC NULLS LAST); CREATE INDEX idx_dest_active ON products (is_active) WHERE is_active = TRUE;

2. Matches Table (optional)

When to use: If the client wants to see cross-platform arbitrage opportunities (e.g., "this Amazon BR product matches this ML product, margin is 15%").

Denormalized: Uses source_key instead of internal IDs — works across database restarts and ID remaps.

ColumnTypeConstraintsDescription
idSERIALPKAuto-increment
source_key_aVARCHAR(150)NOT NULLFirst product: platform:platform_id
source_key_bVARCHAR(150)NOT NULLSecond product: platform:platform_id
confidenceNUMERIC(3,2)NULL0.90 = high, 0.70 = medium, 0.50 = low
margin_pctNUMERIC(6,2)NULLEstimated profit margin %
match_methodVARCHAR(20)NULLimage / title / combined
synced_atTIMESTAMPDEFAULT NOW()When this match was last pushed

CREATE TABLE

CREATE TABLE matches ( id SERIAL PRIMARY KEY, source_key_a VARCHAR(150) NOT NULL, source_key_b VARCHAR(150) NOT NULL, confidence NUMERIC(3,2), margin_pct NUMERIC(6,2), match_method VARCHAR(20), synced_at TIMESTAMP DEFAULT NOW(), UNIQUE (source_key_a, source_key_b) ); -- FK to products (via source_key) ALTER TABLE matches ADD CONSTRAINT fk_match_a FOREIGN KEY (source_key_a) REFERENCES products(source_key); ALTER TABLE matches ADD CONSTRAINT fk_match_b FOREIGN KEY (source_key_b) REFERENCES products(source_key); CREATE INDEX idx_match_a ON matches (source_key_a); CREATE INDEX idx_match_b ON matches (source_key_b); CREATE INDEX idx_match_conf ON matches (confidence DESC);

3. Sync Logic

Generate source_key

The source_key is always platform:platform_id.

  • amazon_br:B0CFYR1VHK
  • amazon_us:B09541P9WH
  • ml:MLB53979422

Upsert product (PostgreSQL)

INSERT INTO products ( source_key, platform, platform_id, title, price, currency, url, image_urls, sales_30d, review_count, review_avg, category_l1, category_l2, category_l3, is_active, synced_at ) VALUES ( 'amazon_br:B0CFYR1VHK', 'amazon_br', 'B0CFYR1VHK', 'Kit 6 Pares Meias Puma', 75.40, 'BRL', 'https://amazon.com.br/dp/B0CFYR1VHK', ARRAY['https://...jpg'], 5000, 120, 4.5, 'Moda', 'Meias', 'Cano Médio', TRUE, NOW() ) ON CONFLICT (source_key) DO UPDATE SET title = EXCLUDED.title, price = EXCLUDED.price, sales_30d = EXCLUDED.sales_30d, review_count = EXCLUDED.review_count, review_avg = EXCLUDED.review_avg, is_active = EXCLUDED.is_active, synced_at = NOW();

Delta sync query (what to push)

-- Run this on ArbitLens DB to find changed products SELECT * FROM products WHERE last_updated > '2026-06-23T00:00:00' -- last sync timestamp AND is_active = TRUE;
